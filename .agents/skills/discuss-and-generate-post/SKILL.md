# 技能：讨论并生成新博文

## 什么时候使用本技能

- 当你说“想写一篇关于 X 的文章/博文”，并希望 AI 从想法到定稿全程协作时。
- 当你希望先和 AI 碰撞灵感、讨论结构，而不是“一句指令直接生成整篇长文”时。
- 不用于简单改错别字、小幅润色或对现有文章的微调（那可以直接在对应文件上操作）。

## 目标 / 非目标

### 目标

- 帮助你**先澄清与收敛主题**（你真正想说什么、给谁看、希望读者带走什么）。
- 在充分讨论后，产出一份你确认过的大纲。
- 基于大纲写出中文初稿，保留你的个人表达风格，并在需要你补充个人经历或观点处明确标记。
- 通过数轮协作润色，让结构、逻辑与表达到达你认可的“可以发布”水位。
- 生成最终 Markdown 文本（含 front matter），并写入 `content/post/<slug>.md`。

### 非目标

- 不在你未确认主题/角度时直接生成整篇文章。
- 不在你未确认大纲时擅自决定文章结构。
- 不自动进行发布或部署（流程只到将文章写入内容文件为止）。

## 输入 / 输出

### 输入（你需要提供）

- **基础信息**（可以很粗）：你大致想写的话题或最近在思考的事情，例如“最近迁移 Nix 的一些体验”。
- **期望调性**（可选）：技术向教程 / 经验分享 / 随笔 / 故事化等。
- **目标读者**（可选）：如“给开发者看”“给非技术朋友看”“主要是写给未来的自己”等。
- **已有素材**（可选）：草稿段落、笔记、链接等。

### 输出（你将得到）

- 一份经过你确认的文章大纲。
- 一份在此基础上生成、可多轮打磨的初稿与润色后正文。
- 一份包含 front matter + 正文的 Markdown 文本，写入 `content/post/<slug>.md`。

## 工作流（按顺序执行）

> 任何时候，如果你表示“还没想好”或“需要换个方向”，AI 应该回到更上游的步骤，而不是继续往下机械推进。

### 1. 碰撞灵感与讨论

1. AI 先向你确认以下要点（如果你已在对话中说明，可复述并确认）：  
   - 你现在最想写的主题/话题是什么？  
   - 你为什么现在想写这篇文章（动机）？  
   - 你心中预期的读者是谁？  
   - 你希望读者读完后带走的 1–3 个核心点是什么？  
2. 基于你的回答，AI 用 2–3 种不同角度的表述，帮助你看到可能的切入方式（例如偏“故事线”/偏“教程”/偏“反思”）。  
3. 你对这些角度进行选择或混合，直到你觉得“对，差不多是这个感觉”，再进入下一步。

### 2. 生成并确认大纲

1. AI 根据已确认的主题与角度，给出一份结构清晰的大纲，例如：  
   - 引子（为什么现在写这个）  
   - 背景/问题  
   - 解决思路或经历过程  
   - 若干关键节点的细节  
   - 总结与后续打算/反思  
2. 你可以对大纲中的章节进行增删改、合并或重排。  
3. AI 根据你的反馈更新大纲，直到你明确表示“这个大纲可以了，可以按这个写”。

### 3. 按大纲写初稿

1. AI 以确认版大纲为唯一结构来源，自上而下生成中文初稿。  
2. 对于需要你补充个人经历、具体数据或敏感判断的地方，AI 使用明显标记（例如 `TODO：这里请你补充自己的具体故事`）提示，不擅自替你编造细节。  
3. 生成初稿后，AI 用简短要点总结每一节写了什么，方便你快速扫一眼结构是否符合预期。

### 4. 润色与局部打磨

1. 你指出希望修改的部分：  
   - 可以是某几段、某个小节，也可以是整篇的语气、长度等。  
2. AI 优先对你明确点名的部分进行调整，避免无谓地重写其他你已认可的段落。  
3. 如有需要，AI 可以提出具体修改建议供你选择（例如“版本 A 更简练，版本 B 更细节”）。

### 5. 整体检查

1. AI 主动对全文做一次“编辑视角”的检查，包括但不限于：  
   - 标题是否准确反映文章内容。  
   - 段落间是否存在逻辑跳跃或重复。  
   - 术语使用是否前后一致。  
   - 引用的命令/代码片段是否合理且语法正确。  
2. AI 将检查结果与建议以列表形式给出，由你决定是否采纳并进行最后一轮修改。

### 6. 定稿与写入文件

1. 你明确表示“这版可以作为定稿”。  
2. AI 与你确认以下 front matter 信息：  
   - `title`：文章标题（中文）。  
   - `slug`（可选）：用于 URL 的短标识；如果你没有特殊偏好，AI 可以根据标题给出英文或拼音建议。  
   - `date`：写作日期。  
   - `draft`：是否先以草稿形式存在（例如第一次发布前可以设为 `true`）。  
   - 可选：`tags`、`series` 等。  
3. AI 基于本仓库现有文章的 front matter 格式，生成完整 Markdown 文本（front matter + 正文）。  
4. AI 将文章写入 `content/post/<slug>.md`（如果没有显式 slug，则根据标题生成合适的文件名）。  
5. AI 提醒你在本地运行 `hugo server -D`，检查文章在页面上的排版、目录与链接。

## 验证方式（DoD）

- [ ] 你明确表示对“主题与角度”满意。  
- [ ] 你明确表示对“大纲”满意。  
- [ ] 你明确表示某一版正文可以作为定稿。  
- [ ] 新文件已写入 `content/post/`，front matter 字段完整，`hugo` 构建无报错。  
- [ ] 通过 `hugo server -D` 可以在本地预览到该文章内容且显示正常。

## 失败处理

- 如果在大纲阶段你始终觉得“不对劲”：  
  - AI 应主动引导回到“碰撞灵感”步骤，重新讨论你真正想说的东西，而不是微调现有大纲的小标题。  
- 如果你觉得 AI 写出的内容“太不像你自己”：  
  - 请直接描述你平时写作的特点（例如更口语/更技术/更克制），AI 应在后续轮次中模仿与收敛，而不是继续维持同一语气。  
- 如果在写入文件后 `hugo` 构建失败：  
  - 优先检查 front matter 语法（如 TOML/YAML 的引号、日期格式）  
  - 必要时由 AI 根据报错信息协助你逐行排查并修正，直到 `hugo` 正常通过构建。
